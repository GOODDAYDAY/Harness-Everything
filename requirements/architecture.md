# Architecture

## Philosophy

Harness-Everything exists to let a single LLM autonomously improve a codebase over many cycles, without human supervision. Three principles shape every design decision:

1. **The LLM is the only intelligence; the harness is plumbing.** The framework never decides *what* to change -- it provides tools, enforces safety boundaries, and manages the execute-verify-commit loop. All creative and analytical work belongs to the LLM.

2. **Config is data, code is generic.** Project-specific content -- missions, prompts, paths, API endpoints -- lives in JSON config files, never in Python source. This means the same harness binary serves any codebase without modification.

3. **Safety gates are non-negotiable.** An autonomous agent that can write arbitrary code to disk is dangerous. Every file access goes through path security checks (traversal, homoglyphs, null bytes, symlinks). Every commit is gated by syntax, import-smoke, and static-analysis hooks. These gates are not optional -- they are the reason unattended operation is feasible.

## Architecture Overview

### Layered structure

```
main.py                          Entry point -- parses config, calls asyncio.run()
    |
    v
AgentLoop (harness/agent/)       Outer loop: cycles, notes, commit, evaluate
    |
    +-- LLM (harness/core/)      Inner loop: tool-use dialogue with the model
    |       |
    |       +-- ToolRegistry     Dispatches tool calls, normalises param aliases
    |       |       |
    |       |       +-- Tool*    40+ tools (file ops, search, git, bash, AST...)
    |       |
    |       +-- Pruning          Compresses conversation history to stay in context
    |
    +-- Hooks (harness/core/)    Post-cycle verification (syntax, import, static)
    |
    +-- DualEvaluator            Two isolated LLM evaluators score each cycle's diff
    |   (harness/evaluation/)
    |
    +-- ArtifactStore            Hierarchical run/cycle output persistence
        (harness/core/)
```

### Execution flow

Each run consists of up to `max_cycles` cycles. Within each cycle:

1. **Prompt assembly.** The system prompt is built from the mission string plus the tail of `agent_notes.md` (capped at `max_notes_cycles` entries). This gives the LLM cross-cycle memory without unbounded prompt growth.

2. **Tool-use dialogue.** `LLM.call_with_tools()` runs a connected tool-use loop for up to `max_tool_turns` turns. The LLM reads files, searches, edits, runs shell commands -- whatever it judges necessary. The conversation is pruned mid-flight when it approaches context limits.

3. **Verification hooks.** After the tool loop finishes, `SyntaxCheckHook`, `ImportSmokeHook`, and `StaticCheckHook` run in sequence. Any gating hook failure blocks the commit.

4. **Commit.** If hooks pass and `auto_commit` is enabled, changed files are committed. Commit messages are generated by the LLM during its tool dialogue.

5. **Evaluation.** `DualEvaluator` runs two isolated LLM calls (basic + diffusion) against the cycle's git diff. Scores are logged and injected into subsequent cycles so the agent sees quality trends.

6. **Meta-review.** Every `meta_review_interval` cycles, a separate LLM call analyses score trends and git history, producing strategic guidance for the next batch.

7. **Termination.** The cycle ends when the LLM emits "MISSION COMPLETE" or "MISSION BLOCKED", or when `max_cycles` is reached. In `continuous` mode, "MISSION COMPLETE" is ignored.

### Self-improvement loop (server deployment)

```
Agent runs N cycles --> commits --> push --> tag
                                              |
GitHub Actions sees tag --> SSH to server --> smoke test --> deploy --> restart
                                                                        |
                                              Agent runs N more cycles...
```

Python loads modules once at startup. When the harness edits its own code, restarting the process is the only way to apply changes. The push-tag-deploy-restart cycle solves this: each generation of the harness produces the next generation's code, then yields control to the CI system which validates and deploys it.

## Key Decisions

### Single-agent, no pipeline

Earlier versions (V1-V4) used multi-phase pipelines with separate proposer/debater/executor roles. V5 collapsed this into a single LLM with full tool access. The motivation: multi-agent coordination overhead exceeded the value of specialisation. A single agent with 40+ tools and persistent notes achieves better results with simpler code.

### Dual-isolated evaluation

Two evaluators (basic correctness + diffusion/system-impact) run independently and never see each other's output. This prevents groupthink -- if both evaluators converge on a score, the signal is stronger than a single evaluator's judgment. The 60/40 weighting (basic/diffusion) reflects that correctness matters more than architectural elegance.

### Conversation pruning over sliding windows

Long tool-use dialogues can push past 200K characters. Rather than using a sliding window (which loses early context), the LLM module estimates conversation size after each tool-result batch and prunes the oldest tool results when a threshold is exceeded. This preserves the conversation structure while keeping token costs bounded.

### Parameter alias normalisation

LLMs frequently hallucinate parameter names (`file_path` instead of `path`, `old_string` instead of `old_str`). Rather than wasting tool turns on error-retry loops, the `ToolRegistry` normalises common aliases before dispatch. This is a pragmatic concession: the LLM is imperfect, and the harness should absorb predictable imperfections silently.

### Path security as a hard boundary

Every file-accessing tool calls `_check_path()` before any I/O. This validates against directory traversal, null bytes, Unicode homoglyphs, and symlink escapes. The check is mandatory -- there is no `--force` flag, no admin override. The workspace boundary is the security perimeter, and it is enforced at the tool layer, not the LLM layer.

### Config-driven hook composition

Verification hooks are selected by name in the config (`cycle_hooks: ["syntax", "static", "import_smoke"]`). Hooks declare whether they gate commits. This lets operators tune the safety/speed tradeoff per project: a prototype might skip static analysis, while a production codebase demands all three.

### Persistent notes over conversation history

Cross-cycle memory lives in `agent_notes.md` on disk, not in conversation history. Each cycle starts a fresh conversation with only the mission + recent notes. This avoids context-window exhaustion across long runs and makes the agent's memory inspectable and editable by operators.

## Extension Guide

### Adding a new tool

1. Create `harness/tools/my_tool.py` with a class that extends `Tool`.
2. Implement `name`, `description`, `input_schema()`, and `async execute(config, **params) -> ToolResult`.
3. If the tool accesses the filesystem, set `requires_path_check = True` and call `self._check_path()` before any I/O.
4. Register in `harness/tools/__init__.py` by adding the class to `_ALL_TOOLS`.

The tool is now available to the LLM in all subsequent runs. No config changes needed -- tools are auto-discovered from the registry.

### Adding a new verification hook

1. Subclass `VerificationHook` in `harness/core/hooks.py`.
2. Implement `name`, `gates_commit` (whether failure blocks commit), and `async run(config, context) -> HookResult`.
3. Register the hook name in `_build_hooks()` inside `agent_loop.py`.
4. Add the hook name to `cycle_hooks` in your config JSON.

### Supporting a new LLM provider

The harness uses the Anthropic SDK's message format. Any provider that exposes an Anthropic-compatible endpoint works out of the box -- set `base_url` and `api_key` in config. DeepSeek, AWS Bedrock (via LiteLLM prefix), and custom gateways all work this way. No code changes needed for provider switching.

### Adding a new evaluation dimension

The `DualEvaluator` calls two prompt templates (basic + diffusion) from `harness/prompts/dual_evaluator.py`. To add a new evaluation perspective:

1. Add a new prompt template in `harness/prompts/dual_evaluator.py`.
2. Extend `DualScore` with the new dimension.
3. Update `DualEvaluator` to run the additional evaluation call in parallel.
4. Adjust the `combined` score weighting.

### Customising per-project behaviour

All project-specific tuning goes in the config JSON:

- `mission` -- what the agent should work on.
- `extra` -- arbitrary key-value pairs injected into the system prompt (coding conventions, domain glossary, focus areas).
- `bash_command_denylist` -- shell commands the agent cannot run.
- `allowed_paths` -- filesystem boundaries beyond the workspace root.
- `syntax_check_patterns` -- which files the syntax hook validates.
- `import_smoke_modules` -- which modules the import hook exercises.
